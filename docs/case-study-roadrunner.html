Roadrunner CRM — the design-to-code loop that finally closed.

The Problem

Two problems I'd spent twelve years watching from the design side. One build to finally solve both.

Problem one — design and code were always two languages pretending to speak to each other. Figma handed off, engineering rebuilt visually-close-enough, tokens drifted, four sprints in the two files stopped resembling each other.

Problem two — the one every CRM I shipped fought chronically at Madison Resources recruiting and PayFactors compensation — was dirty data. Duplicate records. Stale contacts. Sources nobody could trace.

On this build I closed the design-to-code loop with Acme and Figma MCP, and I wired real AI suggestions that prevent dirty data on entry — duplicate detection, auto-populate from live corporate registries, attribution per field. The company-intake surface is where this product earns its keep: a real company name fires SEC EDGAR, GLEIF, Companies House, OpenCorporates, and Wikidata in parallel and the matching record populates before the user finishes typing. Record creation went from minutes to seconds. This is not a Figma prototype with simulated AI states — it is a live working product, AI firing against real APIs, open for any hiring manager to click through and try to break.

Then the build kept going. Supabase replaced browser-local state with a real Postgres backend. Google OAuth plus Gmail sync pulled real inbox threads onto every contact timeline. Fourteen contextual tours wire every flow to a help-panel walkthrough. What started as a five-day portfolio sprint has kept landing features the industry charges for.

Research · Career Inquiry

Dirty data was the problem at Madison Resources. Dirty data was the problem at PayFactors. Every HR team lives inside it.

Recruiters at Madison Resources spent hours every week reconciling duplicate candidate records, stale contact info, and fragmented source attribution. Comp analysts at PayFactors fought the same fight against outdated company data and records that said different things in different systems. Different industries, same chronic failure mode.

Dirty data is the problem HR and comp professionals have asked for help with for as long as I've designed for them — and it is the exact problem Roadrunner's AI features are wired to solve.

So when my first pass shipped a fake duplicate detector querying a hardcoded name pool and labeling the fake results "Clearbit verified" — I was replicating the problem I'd spent twelve years trying to fix. The rebuild wired real AI suggestions on every intake surface: type a name, get a duplicate check before you save; type a company, get auto-populate from live public records with provider attribution per field; drop a resume, get parsed fields prefilled and the original attached. Clean record creation went from minutes to seconds.

Key Finding

The gap was never about tools. It was that design and code were two translations of each other.

On a normal project, a designer updates a color token in Figma. An engineer reads the hex. Types it into a stylesheet. Maybe misreads it. Maybe uses the old value because the variable was already defined. Every token is a handoff with a chance of loss. Multiply by every component, every state, every sprint. That is where the gap lives. I brought this framing to the Acme build before I drew a single Figma frame.

The close

Acme tokens are defined once. Figma variables and Tailwind CSS custom properties reference the same values. Figma MCP reads the Figma file. Claude Code generates React using Tailwind classes that resolve to those same tokens. A change in one surface shows up in the other without translation.

"Design linked to code. Code linked to design. First time in twelve years that loop closed on anything I built."

The AI Layer

Real APIs. Free tier. Honest about the scope.

Every company lookup runs against a live corporate registry — SEC EDGAR for public filings, GLEIF for legal entity identifiers, Companies House for UK filings, OpenCorporates for the global aggregate, Wikidata for the knowledge graph, Clearbit for enrichment. The suggestions that populate the form carry a source badge back to the record they came from. That's the durable, trustworthy core of this build.

For individuals, the public layer is intentionally thin: Wikidata and ORCID cover public figures and researchers, GitHub covers developers who opt in. I will not wire a paid people-data aggregator into a portfolio demo — the value of "trusted, traceable data" evaporates the moment the source is a black-box scraper. A resume parser, a LinkedIn URL field, and the CRM's own records carry the person side; public registries carry the company side.

The Backend Turn

The sandbox got a real database, then a real inbox. Portfolio demo stopped pretending.

For the first week Roadrunner stored everything in the browser — Zustand persist, per-visitor localStorage, reset-on-demand. That was honest for a portfolio sandbox. It was not honest for "real working CRM." The database turn made multi-session state durable: Supabase Postgres with Row Level Security, server-side migrations via a signed exec_sql installer, every table keyed to auth.uid(). Every contact, deal, list, and document now persists across sessions and browsers — and the seed data stays isolated in browser-local land so the demo still feels instant on first visit.

Then the inbox turn. Google OAuth with refresh-token capture, a Gmail sync worker that resolves senders to contacts by email match, and a unified timeline on every contact Overview. The Activity card composer sends from the CRM and lands in Gmail Sent — same as HubSpot, Salesforce, Close, Attio, Folk. A top-sender import surface turns the first sync into onboarding: people you already email are already contacts, before you open the grid. This is the capture layer the four biggest CRMs build their paid tiers on.

The competitive read

On email capture — OAuth connect, message-to-contact matching, send-from-CRM, top-sender import — Roadrunner is at parity with Attio and Folk, ahead of monday and basic Pipedrive. On email activation — templates, open/click tracking, sequences, bulk merge, meeting-scheduler in compose — it is behind every major, which is honest scope for an eight-day build with a real backend.

Design Decisions

Ten decisions. Each one a direct response to something AI generated that experience had to correct — or something users have asked for in every product I've shipped.

Decision 01

Kill the fake data pool. Rebuild every AI path against eleven real public sources.

Finding The first build queried a 2,847-entry hardcoded name pool and labeled the result "Clearbit verified." The attribution was a string constant, not an API call.
Insight This wasn't a data bug — it was a trust bug. Every AI feature makes an implicit claim when it fires. The claim has to be inspectable or the demo dies on the first real name.
Design Server-side fan-out across eleven public providers. Provider badges render on every suggested field. Any claim can be traced in one click.
Decision 02

Sticky Actions column on every grid. Inline icons over a three-dot menu.

Finding Claude's first pass on the Placements grid hid edit and delete behind a three-dot overflow menu — two clicks and a right-edge scroll on narrow viewports to reach either.
Insight Overflow menus earn their place at five or more actions. With two, they're extra clicks with no payoff. Recruiters at Madison Resources asked me to remove this exact pattern years ago.
Design Edit and delete as inline icons, right-edge-pinned across every grid. Hover-reveal keeps rows quiet. Matches Salesforce, Linear, and HubSpot defaults.
Decision 03

Private filter: brand blue, not danger red.

Finding First pass shipped the Private filter pill in red because "private" felt weighty. Every other filter in the bar was brand blue.
Insight Red is reserved for destructive. Using it for a category filter teaches users to flinch at a safe control and breaks the semantic color contract elsewhere in the app.
Design Brand-blue fill, matching Favorites and the type filter. The filter bar reads as one control group instead of a safe-controls-and-one-warning cluster.
Decision 04

Column pinning — persisted, always visible.

Finding First pass hid pin icons behind column-row hover. Users who wanted pinning couldn't find it.
Insight Hover-reveal is right for rare controls. Pinning is a power feature users look for by name — Excel freeze-pane muscle memory is thirty years old. Hiding it buries the benefit they came for.
Design Pin buttons always visible in the Columns dropdown. State persists across refresh via Zustand persist middleware. Left and right edges both supported.
Decision 05

Tabbed help panel. Drop the count badge.

Finding The first help panel was eleven tour cards stacked in a 600-pixel side panel. Most tours sat below the fold and got skipped.
Insight A scrollfest in a fixed-height surface is a discoverability failure, not a cosmetic one. A count badge without units is decoration, not information.
Design Pill-segmented Tips / Tours / Resources tabs — matching the List/Card and All/Organizations/People pills elsewhere. Current-page tour pinned at top as "Recommended for this page." No count.
Decision 06

Tours live in the help panel — always available, not a one-time install cutscene.

Finding Most products ship guided tours that fire once on first login and then disappear forever. Users who needed help later — a month in, returning after a break, training a new hire — had no way to find the walkthrough again.
Insight Help is most useful when the user is already frustrated, not when they're fresh. Burning tours on install wastes them on the moment users need them least. This came from watching support-ticket patterns at PayFactors and Madison Resources — tickets spiked in week three, not week one.
Design Fourteen contextual tours live permanently in the help panel, reachable from any page, filtered to the current surface first. Every tour can be re-run any time. No cutscenes, no dismiss-forever, no buried "restart tutorial" setting. The help icon is the entry point; the tour is always behind it.
Decision 07

Light and dark mode, defined at the token layer so every surface retints for free.

Finding Light and dark mode is the single most-requested feature across every CRM I've shipped — financial services, HR staffing, compensation analytics. Users ask for it in week one. It ships in year three because teams retrofit it component by component.
Insight Dark mode isn't a styling pass. It's a token problem. If colors are defined per-component, you retrofit forever. If colors are defined as semantic tokens (surface, text-primary, border) in an Acme-style system, dark mode is one theme swap away.
Design Every Acme color is a semantic token mapped to a light and a dark value. The theme switch in Settings flips a root-level data attribute; every surface in every module retints instantly. No per-component work, no missed states. Shipped on day one, not year three.
Decision 08

Toast confirmations with Undo on every destructive action.

Finding Initial delete handlers fired silent success. Users re-clicked delete because they were unsure the first click had registered.
Insight Silent success is disorienting. Gmail, Slack, and Figma anchor destructive actions on toast-with-Undo because reversibility reduces the fear tax on irreversible controls.
Design Four-severity toast system. Snapshot-and-restore Undo on every contact, deal, and candidate delete. Caught the missed contact-delete case when I asked, "If I delete a contact will I see a toast?"
Decision 09

Real Postgres backend, then real Gmail sync — stop pretending to be a sandbox.

Finding v1 stored every contact, deal, and list in browser-local Zustand. Open the app in a second browser and you got an empty CRM. Refresh on the wrong device and your session was gone. "Real working CRM" was stretching it.
Insight Durability is the line between portfolio prototype and product. And email is where a CRM earns its keep — HubSpot, Salesforce, Close, Attio, and Folk all built their paid tiers on inbox integration. A CRM that couldn't see a sender's email thread was missing the capture layer the whole category charges for.
Design Supabase Postgres with Row Level Security on every table. A signed exec_sql installer for migrations so schema pushes don't need a raw DB password in CI. Google OAuth with refresh-token capture. A Gmail sync worker that resolves senders to contacts and stitches threads onto every Overview timeline. A top-sender import surface that turns the first sync into onboarding. Everything AI-generated; every integration cited against how the majors do it.
Decision 10

Tours must start at the action that triggers the flow — and the navigation effect has to outlive the help panel that launched it.

Finding First three tour rewrites all failed the same way. The Person-creation tour started on the form page, so step 1 had no "+ New Contact" button to highlight. The Lists tour never showed a list view. The Company tour flashed step 2 and vanished — clickTarget was firing on the card, which navigated the router before the spotlight rendered. Then, when I fixed those, steps 4–6 had nothing to highlight because the form page never loaded — the per-step navigateTo effect lived in the help panel, which auto-closes when a tour starts, killing every navigation after step 1.
Insight A guided tour is a sequence of promises: "click this, then this, then this." The component that owns the sequence must be the component that lives the whole sequence. The help panel is ephemeral; the tour overlay is persistent. Ownership had to move. And clickTarget is a contract about what happens when the spotlight mounts — safe for in-place actions (open a dropdown, a slide panel, a picker portal) but fatal when the click navigates, because the navigation kills the bubble.
Design Rebuilt every tour to start at the entry-point action (/contacts spotlighting "+ New Contact," not the form page). Moved the navigateTo effect out of HelpPanel.tsx and into ActiveTourOverlay.tsx, which stays mounted for the whole tour. Added a URL-plus-querystring comparison because usePathname() drops the query and would have put the lists tour into an infinite navigation loop. Codified the clickTarget rule in code comments so the next person hitting this doesn't lose two hours to it. Four iterations, four rejections, one principle that will outlive this build: the overlay owns the navigation because the overlay is the only thing that lives as long as the tour.
What I Deliver to Engineering

Because the loop closes, I hand off deliverables engineering can consume — not specs to translate.

A senior designer who has closed the design-to-code loop doesn't move stories from "in design" to "in dev." They move them from "in design" to "merged" — because the artifacts engineering receives are already the artifacts engineering would build. That's the difference between one sprint and three.

Without the loop

What engineering gets today on most projects

  • Figma file with redlines to measure
  • PNG exports of states
  • Hex codes copy-pasted into CSS
  • Slack threads clarifying what a component means
  • Design drift discovered three sprints later
  • QA tickets retrofitting accessibility
  • Coordination meetings to align tokens
With the loop

What engineering gets from me

  • Running token library — install and import
  • Live React components next to Figma frames
  • Figma-to-code prop mapping, one page per component
  • Validation rules authored on the design side
  • Accessibility baked into the system
  • AI-generated first drafts with judgment applied
  • One master doc, not thirty Slack threads
Deliverable 01
A running design system, not a style guide
Acme is a working token library. Figma variables and Tailwind CSS custom properties wired to the same source of truth. Engineering installs, imports, and ships.
Deliverable 02
Live component references, not static specs
Every component lives in both Figma and a running React surface. Engineers click from the handoff to the rendered component with exact props and states.
Deliverable 03
Grid pattern library from 12 years of CRMs
The grid toolbar — saved views, pinning, density, zebra, sticky actions, per-column filters — shipped as one reusable SharedDataGrid. One component to maintain, eight grids to use it.
Deliverable 04
Tokens that survive refactors
Because tokens are defined once and referenced in both surfaces, engineering can rename, restructure, or reorganize without a coordination meeting. Figma updates when tokens do.
Deliverable 05
Working prototypes instead of redlines
When I propose a pattern, I can ship a working prop-typed React component alongside the Figma frame. Engineering reviews behavior and props, not pixel measurements.
Deliverable 06
Figma-to-code mapping documents
One page per component mapping Figma props to React props and CSS variables. No guessing, no "what does size=lg mean in the real file." Engineering reads, writes glue, ships.
Deliverable 07
AI-assisted first drafts — with judgment applied
For rare patterns, I generate a first-draft React component with Claude Code and hand it to engineering with naming cleaned up, states verified, accessibility checked, tokens wired.
Deliverable 08
Validation rules as design, not copy
Intake forms ship with required fields, format checks, error messages, and disabled states defined as props. Engineering implements the glue; rules live where the form lives.
Deliverable 09
Accessibility at the system level
Color contrast, focus states, keyboard navigation, and screen-reader labels are baked into Acme components. Use the button, get the a11y. No retroactive QA tickets.
Deliverable 10
One artifact, not thirty Slack threads
The Acme master document — foundations, components, tokens, patterns, Figma-to-code mapping, grid library, validation, accessibility — in one navigable reference. Engineering bookmarks one URL.

What I'd Do Differently

Honest, one paragraph.

I'd wire the real providers from day one instead of letting the fake pool ship first — the rebuild was instructive but expensive. I'd put validation on every intake surface before writing a single AI feature, not after. I'd ship the Postgres backend alongside v1 instead of letting "browser-local only" be the story for a week; the sandbox framing was convenient for a portfolio sprint and wrong for a CRM. On tours, I'd write the first one by walking through the app as a new user instead of starting mid-flow — four rewrites taught the same lesson I could have gotten for free. The pattern that held best was pairing every Claude-generated decision with a cited precedent from my career. The pattern I'd drop was trusting first-pass toast wiring and first-pass tour wiring to be complete; both missed cases I caught only by running the flow cold.
